## 2. Focus Management — Code Review

### Modals (OCRConfigModal, WordEditDialog, HotkeyHelpModal, ConfirmDialog, ExportDialog)

- `SourceFolderDialog.tsx` uses `autoFocus` on the path-input. When dialog
  opens, focus lands on the text field immediately. `Escape` calls `handleCancel`
  via `onKeyDown`. `Mod+Enter` calls `handleApply`. Verified 2026-05-16 — handler
  code correct at `handleKeyDown` lines 171–183. **No focus trap** — Tab navigates
  the page behind the overlay (known gap, tracked in §8 item 1).
- `WordEditDialog.tsx` uses `autoFocus` on the GT textarea. Escape and Enter
  are handled via `useDialogHotkeys`. Verified 2026-05-16 — bindings confirmed
  in `frontend/src/hooks/useDialogHotkeys.ts`. Same focus-trap gap as SourceFolderDialog.
- `HotkeyHelpModal.tsx` — rendered as a full-screen overlay. Close button
  present; `Escape` bound via `useHotkey("escape", ...)` (line 83). `?` binding
  confirmed line 81. Verified 2026-05-16 — both bindings present in source.
  Focus trap absent (known gap). Tab order inside modal table may reach background
  — blocked on browser walk but deprioritised given focus-trap follow-on.
- `ConfirmDialog.tsx` — `autoFocus` on Confirm button (line 72). No explicit
  keyboard bindings. Escape/Enter behavior depends on browser's default button
  focus handling. Bug filed: BUG-KBD-4.
- `OCRConfigModal.tsx` — rendered inline with `role="dialog"`. Close button
  present; no `useHotkey` registration for Escape found. Focus trap absent (known
  gap). Mod+, hotkey not wired — Bug filed: BUG-KBD-1.

### Known Gap: No Focus Trap

None of the dialogs implement a focus trap (e.g. `focus-trap-react` library or
manual `onKeyDown Tab` cycling). This is a potential axe-core failure for WCAG
2.1 §2.1.2 (No Keyboard Trap). The axe-core E2E test (`tests/e2e/test_a11y.py`)
validates open pages but does not open modals during the audit.

**Recommendation:** Add focus traps to all four modal dialogs before final
M9.5 sign-off. Low complexity — `focus-trap-react` is a one-line wrapper.
Track as separate issue.

## 4. Tab Order — Project Page

ProjectPage renders (left-to-right, top-to-bottom):

1. `HeaderBar` — breadcrumb, change-project button, page nav buttons, OCR
   Config button
2. `ImageTabsHeader` — Words / Lines / Paragraphs / Matched-Words tabs + mode
   radio buttons
3. `BBoxOverlay` / `PageImageCanvas` — Konva stage (no native tab stop; canvas
   receives keyboard events only when programmatically focused)
4. `TextTabs` — OCR / GT / Plaintext tabs
5. `WordMatchView` — scrollable virtualised list of line cards

Potential gap: the Konva canvas (`<canvas>` element) has no `tabIndex` by
default. Keyboard navigation into the viewport requires Tab to land on the
canvas or a containing div with `tabIndex=0`. Verified 2026-05-16 —
`PageImageCanvas.tsx` line 529 confirms `tabIndex={0}` on the outer wrapper div
(`data-testid="image-viewport"`), and line 297–298 calls `wrapperRef.current?.focus()`
on mount. Canvas is reachable via Tab and auto-focuses on render.
