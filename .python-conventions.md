# Python conventions — pdomain-ocr-labeler-spa

The `writing-python` skill reads this file when it runs. This file contains only project-specific
differences. The plugin defines the procedure, strict baseline, and gate-resolution order.

## Commands that prove completion

- Run the repository gates with `make lint`, `make format`, `make typecheck`, and `make test`.
- Run scoped tests with `uv run pytest tests/AREA -q`, replacing `AREA` with the area you changed.
  Use `-k` when you need to select tests by name. Run the integration-only full suite serially.
- Completion requires this project's print-mode gate and scoped tests. `resolve_gate.sh --run`
  provides a consistent backstop. Hooks are advisory and never prove completion.

## Local PostToolUse opt-in for writing-python

- Claude and Codex load hooks owned by the plugin. Grok first requires
  `hooks/grok/install_grok_hooks.sh` to install its native global hook.
- After the host hook is available, create an **empty, gitignored** file at
  `.claude/writing-python-gate.local`. Do not commit it because the helper refuses a tracked marker.
- The opt-in marker remains active until you remove it. It enables the installed Grok hook and the
  Claude and Codex hooks. Without the marker, the hook does nothing. Silence or exit 2 means the
  result is unknown, not green.
- Claude, Codex, and Grok run Stop only after the same host changes an existing Python file in the
  same repository and session. Each Python-touch ledger entry is one-shot. Bash-only activity stays
  silent.
- Codex loads plugin-owned hooks from `hooks/codex/hooks.json`. Its `systemMessage` warnings are
  visible in the UI or event stream. They are not completion evidence visible to the model. After
  installing or changing hooks, start a new session and approve the changed plugin hook definitions
  in `/hooks`.
- The repository installer removes any legacy inline Codex block after installing the plugin.
  `hooks/codex/install_codex_hooks.sh --rollback` restores that block for recovery only.

## Apply strict checks gradually in brownfield code

- Prefer gradual zones, such as `executionEnvironments` or path-scoped basedpyright, over changes
  across the whole tree.
- Treat pre-existing lint outside the work area as backlog. Do not rewrite the monorepo at Stop time.
- Read the plugin's lazy reference `python-brownfield.md` when this repository is only partially
  strict.

## Project settings the resolver cannot infer

- The first-party import package is `pdomain_ocr_labeler_spa`. Set
  `[tool.ruff.lint.isort] known-first-party` to match.
- This repository enables no optional Ruff rule families beyond the baseline. Add any of
  `T20`/`TRY`/`ISC`/`ERA`/`PLR`/`PLW`/`S` if wanted. Enable `T20` and `print` only if this repository
  has no intentional command-line or print output.
- Add dependency injection calls to Ruff's immutable-call list to avoid B008. For FastAPI
  `Depends()`, use this setting. Add the corresponding immutable call names when using Click.

  ```toml

  extend-immutable-calls = ["fastapi.Depends"]
  ```
